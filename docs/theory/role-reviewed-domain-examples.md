# Role-Reviewed Domain Examples

This note applies the FRAMES role lenses to concrete source domains and local
patterns. It is not a catalog expansion. Its job is to show how a domain example
is reviewed before it becomes an accepted frame, frame pack, or indexed entry.

These reviews are AI-simulated role lenses from `.roles/`, not claims of
external endorsement.

## Review Shape

Use this shape for domain examples:

| Field | Meaning |
|---|---|
| Candidate pattern | The frame, source domain, or local pattern under review. |
| Target use | The work situation this should clarify. |
| Primary job | Status, coordination, momentum, risk, priority, or learning. |
| Story job | Narrative purpose, if any. |
| Claim strength | Current evidence label. |
| Relational transfer | The main source-target structure. |
| Evidence boundary | What still must be checked. |
| Role verdict | Accept, revise, hold, or reject for the stated use. |

## Example 1: Four-Way Stop

| Field | Review |
|---|---|
| Candidate pattern | Four-way stop. |
| Target use | Peer teams need visible turn order around constrained attention or shared dependency. |
| Primary job | Coordination. |
| Story job | Teaching / simulation or repair / empathy. |
| Claim strength | `heuristic`. |
| Relational transfer | Peer actors pause, read order, signal intent, and proceed visibly. |
| Evidence boundary | Check whether parties are peers or whether one owner has authority. |
| Role verdict | Accept for peer coordination; revise or reject for asymmetric authority. |

Role pressure:

| Role | Review |
|---|---|
| Frame Fit Reviewer | Strong because it changes action: pause, read order, signal, proceed. |
| Business Leader | Useful for sequencing shared attention, weak for priority ownership. |
| Evidence Boundary Reviewer | Must name authority and urgency checks. |
| Misuse Risk Reviewer | Do not call another team the obstacle or collision risk. |
| Catalog Structure Reviewer | Good candidate because job, authority model, and failure mode are stable. |

## Example 2: Red / Yellow / Green

| Field | Review |
|---|---|
| Candidate pattern | Red / yellow / green status. |
| Target use | Shared readiness, risk, or confidence reporting. |
| Primary job | Status. |
| Story job | Decision / tradeoff or warning. |
| Claim strength | `heuristic`. |
| Relational transfer | A visible signal changes behavior at agreed thresholds. |
| Evidence boundary | Check thresholds, owner, movement criteria, and confidence. |
| Role verdict | Accept when thresholds are explicit; revise when colors are subjective. |

Role pressure:

| Role | Review |
|---|---|
| Frame Fit Reviewer | Strong only when each color changes a decision or next action. |
| Business Leader | Must distinguish health, progress, confidence, and priority. |
| Evidence Boundary Reviewer | Color cannot replace metric, owner judgment, or threshold evidence. |
| Misuse Risk Reviewer | Red must not become blame; green must not imply zero risk. |
| Catalog Structure Reviewer | Needs fields for threshold, owner, and movement condition. |

## Example 3: Bag Of Chips Conflict Story

| Field | Review |
|---|---|
| Candidate pattern | Person thinks another party is taking their chips, then discovers the ownership story was reversed. |
| Target use | Cross-team conflict where facts, ownership, or perspective may be incomplete. |
| Primary job | Risk / coordination. |
| Story job | Repair / empathy. |
| Claim strength | `locally_observed` until extracted into a reusable conflict-repair frame. |
| Relational transfer | Initial interpretation changes when missing evidence changes the ownership model. |
| Evidence boundary | Check facts, ownership, harm, and repair obligations before treating conflict as misunderstanding. |
| Role verdict | Hold as a story example; revise before catalog acceptance. |

Role pressure:

| Role | Review |
|---|---|
| Frame Fit Reviewer | Useful for slowing judgment, but action cue needs a concrete repair workflow. |
| Business Leader | Helps reduce escalation cost, but must not delay accountability. |
| Evidence Boundary Reviewer | Empathy story does not settle facts. |
| Misuse Risk Reviewer | Do not use it to pressure forgiveness or minimize harm. |
| Expert Storyteller | Memorable because reversal is simple and portable. |
| Novice Reader | Needs plain setup; avoid assuming the reader knows the Mrs. Fields cookie variant. |

## Example 4: Theme Swimlanes

| Field | Review |
|---|---|
| Candidate pattern | Program themes such as `Run One`, `Run Lean`, `Run Fast`, `Run Safe`, or three customer-promise lanes such as more control, more visibility, less effort. |
| Target use | Multiple teams need to align different work into a small number of memorable contribution lanes. |
| Primary job | Priority / coordination. |
| Story job | Vision / destination and decision / tradeoff. |
| Claim strength | `locally_observed`. |
| Relational transfer | A shared promise creates contribution lanes; teams map work into lanes that customer outcomes can recognize. |
| Evidence boundary | Check lane owner, customer outcome, measure, tradeoff, and what work is excluded. |
| Role verdict | Revise into a structured import before cataloging. |

Role pressure:

| Role | Review |
|---|---|
| Frame Fit Reviewer | Strong if lanes change prioritization and sequencing; weak if only slogans. |
| Business Leader | Useful when it clarifies investment lanes, promise, and tradeoffs. |
| Evidence Boundary Reviewer | Needs measures for control, visibility, effort, speed, safety, or unity. |
| Misuse Risk Reviewer | Watch for theme names that force agreement or hide cuts. |
| Catalog Structure Reviewer | Needs a repeatable template: theme, promise, lane, owner, measure, exclusion. |
| Journeyman Practitioner | Valuable if each team can place real work in a lane without a planning workshop. |

## Review Outcomes

| Pattern | Outcome | Next Move |
|---|---|---|
| Four-way stop | Accept for peer coordination. | Keep authority warning visible. |
| Red / yellow / green | Accept with threshold discipline. | Add threshold and movement fields in future metadata. |
| Bag of chips conflict story | Hold as story support. | Create conflict-repair frame with facts, ownership, and repair steps. |
| Theme swimlanes | Revise toward structured import. | Build a theme-swimlane extraction artifact. |

## Method Lessons

- A role-reviewed example can pass for one use and fail for another.
- Claim strength often differs from practical usefulness; a local pattern may
  be valuable while still labeled `locally_observed`.
- The business-leader role should force decision, tradeoff, owner, or customer
  promise clarity.
- The evidence-boundary role should block slogans from becoming proof.
- The misuse-risk role should check whether the frame pressures agreement,
  hides accountability, or turns people into objects.
- The catalog-structure role should convert useful examples into reusable
  fields before index expansion.
